# REVIEW_01_CODEX — 구현 전 제품/기술 설계 리뷰

> 리뷰 기준: 코드 작성/수정 없이 `docs/` 문서 전체를 읽고, MVP 범위·제품 방향·톤앤매너·안전/윤리·계산 로직·티어 시스템·UX·디자인·스토리 카드·기술 설계 관점에서 구현 전 리스크를 점검했다.
>
> 참고: `소맥 견습생`은 브랜드 톤상 예외적으로 허용하는 표현으로 본 리뷰에서 위험 문구 수정 대상에서 제외한다.

## 1. 전체 평가 요약

- 현재 문서 품질: **78/100**
- 바로 구현 가능 여부: **문서 수정 1라운드 후 구현 권장**
- 총평: 제품 방향과 기술 설계의 큰 줄기는 좋지만, 문서 간 충돌과 톤 안전성 문제가 남아 있다.

### 가장 큰 리스크 3개

1. **톤 가이드와 티어 카피가 충돌함**
   - `레전드/전설`은 금지어인데 `t6_legend`, `전설의 숙취술사`, `기록은 전설` 등에 쓰인다.

2. **0 입력/빈 기록 처리 정책이 문서마다 다름**
   - `CALCULATION_RULES`는 0 입력 결과 페이지 허용.
   - `UX_FLOW`는 기록 없으면 메인으로 리다이렉트.
   - `IMPLEMENTATION_ROADMAP`은 0 입력 결과 페이지 정상 표시를 요구.

3. **MVP가 다소 큼**
   - 5종 카드 템플릿, 주간 리포트, 다크모드 자동 대응, PWA 메타, 테스트 하네스, 공유 fallback까지 모두 MVP에 포함되어 초기 구현 난도가 올라간다.

## 2. 문서별 리뷰

### PROJECT_BRIEF

- 잘 된 점
  - 개인 간 리포트 중심, 야장맵과의 차별점, `자랑과 자학 사이` 포지셔닝이 명확하다.
- 문제점
  - 타깃을 대학생으로 두면서 미성년자 배제 장치가 약하다.
- 수정 필요 사항
  - `성인 대학생`을 더 일관되게 명시하는 편이 안전하다.
- 구현 모호성
  - 낮음.
- 안전/톤 리스크
  - `자랑` 중심 표현이 너무 앞서면 과음 인증처럼 보일 수 있다.

### PRD

- 잘 된 점
  - 포함/제외 범위가 잘 나뉘어 있고 로그인, 서버, 랭킹, AI 이미지 생성 제외가 명확하다.
- 문제점
  - MVP 기능이 많다. 5종 카드 템플릿과 주간 카드까지 첫 버전에 넣는 것은 무겁다.
- 수정 필요 사항
  - 카드 템플릿 수, 주간 카드, 다크모드/PWA를 MVP 필수와 확장으로 재분류해야 한다.
- 구현 모호성
  - `기록 수정·삭제(값 0)`와 `빈 기록`의 의미가 불명확하다.
- 안전/톤 리스크
  - `술 마신 직후` 사용은 운전 가능성 암시와 붙을 수 있어 안전 안내 위치가 중요하다.

### UX_FLOW

- 잘 된 점
  - 모바일 흐름, 요일바, 입력, 결과, 카드 흐름이 구현자가 따라가기 좋다.
- 문제점
  - 0 입력/빈 기록 처리 충돌이 크다.
- 수정 필요 사항
  - `0 값으로 저장된 기록`과 `기록 객체 없음`을 구분해야 한다.
- 구현 모호성
  - 자동 저장 상태 dot, 토스트 host, 리다이렉트 타이밍이 기술 문서와 연결되어야 한다.
- 안전/톤 리스크
  - `저장 완료 — 스토리에 올려보세요.`는 괜찮지만, 결과가 높은 티어일 때는 공유 유도 문구가 조심스러워야 한다.

### DESIGN_SYSTEM

- 잘 된 점
  - 헌팅포차/클럽 포스터를 피하려는 방향이 분명하다.
  - 컬러 토큰과 컴포넌트 방향도 꽤 구체적이다.
- 문제점
  - 아이보리+딥그린+라임 조합이 잘못 쓰이면 프리미엄 헬스보다 장식적인 밈 카드로 흐를 수 있다.
- 수정 필요 사항
  - 스토리 카드 템플릿별 실제 구성 밀도를 더 제한해야 한다.
- 구현 모호성
  - 노이즈 텍스처를 SVG filter로 할지 CSS 이미지로 할지 불명확하다.
- 안전/톤 리스크
  - 높은 티어 카드가 시각적으로 멋있어 보이지 않게 하는 규칙은 더 강하게 필요하다.

### TONE_GUIDE

- 잘 된 점
  - 금지 표현, 버튼 문구, 안전 안내가 잘 정의되어 있다.
- 문제점
  - 금지어와 실제 티어 카피가 충돌한다.
  - `축하드립니다`는 과음 맥락에서 위험하다.
- 수정 필요 사항
  - 금지어 사전과 예시 카피를 같은 기준으로 정리해야 한다.
- 구현 모호성
  - 금지어 테스트 범위가 `tierMessages`만인지 전체 UI 문구인지 불명확하다.
- 안전/톤 리스크
  - `자랑은 가능`은 브랜드 핵심이지만 너무 반복되면 과음 인증처럼 보일 수 있다.

### CALCULATION_RULES

- 잘 된 점
  - 상수, 공식, 반올림, 입력 방어가 매우 좋다.
  - 테스트 가능성이 높다.
- 문제점
  - 회복시간이 사용자에게 `술이 깬 시간`으로 오해될 수 있다.
- 수정 필요 사항
  - `회복시간` 대신 `간 처리 추정 시간` 또는 `알코올 처리 추정 시간` 같은 표현 검토가 필요하다.
- 구현 모호성
  - 소주/맥주 입력이 정수만인지 소수 허용인지 불명확하다.
  - 소주 0.5병 같은 현실 입력을 막을지 결정해야 한다.
- 안전/톤 리스크
  - 운전 가능 여부와 무관하다는 문구를 회복시간 옆에도 붙여야 한다.

### TIER_SYSTEM

- 잘 된 점
  - 당일/주간 분리, 경계값, 게임화 금지가 명확하다.
- 문제점
  - `t6_legend`, `전설의 숙취술사`, `기록상은 전설`은 금지어와 충돌하고 과음 미화 위험이 있다.
- 수정 필요 사항
  - 높은 티어명 전체 재검토가 필요하다.
- 구현 모호성
  - 20.0과 20.1 사이 같은 소수 경계가 테스트는 가능하지만 UX상 설명은 어색하다.
  - `<=20`, `>20` 방식으로 정의하는 편이 낫다.
- 안전/톤 리스크
  - `야장 정규직`은 재미는 있지만 소속감/성취감처럼 보일 수 있다.
  - `소맥 견습생`은 이번 리뷰에서 예외 허용 표현으로 둔다.

### STORY_CARD_SPEC

- 잘 된 점
  - 9:16, 1080×1920, safe area, 공유 fallback이 명확하다.
- 문제점
  - MVP 5종은 과하다.
  - 템플릿별 정보 우선순위도 아직 추상적이다.
- 수정 필요 사항
  - MVP는 2종, 확장 3종으로 나누는 것을 권장한다.
- 구현 모호성
  - 당일 카드와 주간 카드에서 사용 가능한 템플릿 구분이 불명확하다.
- 안전/톤 리스크
  - `tpl_warning`이 너무 멋진 경고 포스터처럼 보이면 역효과가 날 수 있다.

### TECHNICAL_PLAN

- 잘 된 점
  - 폴더 구조, 타입, LocalStorage 키, 테스트 계획이 구현 친화적이다.
- 문제점
  - `CustomBackgroundUploader.tsx # stub`는 MVP 미구현 기능인데 폴더 구조에 들어가 있다.
- 수정 필요 사항
  - stub도 제거하거나 확장 문서에만 존재하도록 옮기는 편이 낫다.
- 구현 모호성
  - 모든 페이지 Client Component 기본은 단순하지만 Next App Router 관점에서 메타/레이아웃과 경계가 더 명확해야 한다.
- 안전/톤 리스크
  - safety-copy 테스트가 금지어와 고정 안전문구의 `운전`을 어떻게 처리할지 정의해야 한다.

### IMPLEMENTATION_ROADMAP

- 잘 된 점
  - 단계별 완료 기준이 있어 구현 세션 분할이 좋다.
- 문제점
  - 3단계 오타: `금지어 테스트 grin.`
- 수정 필요 사항
  - 0 입력 결과 정책, 카드 템플릿 수, 실기기 QA 범위를 앞 단계에서 확정해야 한다.
- 구현 모호성
  - `0개 테스트로 통과`와 이후 `30개 이상 테스트` 기준은 괜찮지만, 테스트 데이터 기준이 문서 수정 후 동기화되어야 한다.
- 안전/톤 리스크
  - 금지어 테스트 통과를 Done Criteria로 둔 점은 좋다.

## 3. 치명적 이슈 목록

### 이슈명: 금지어와 티어명 충돌

- 심각도: **Critical**
- 해당 문서: `TONE_GUIDE`, `TIER_SYSTEM`
- 문제 설명: `레전드` 금지인데 `t6_legend`, `전설의 숙취술사` 사용.
- 왜 문제인지: 자동 테스트 실패 또는 과음 미화 인식 발생.
- 수정 제안: `t6_legend`를 `t6_redzone` / `t6_overlimit` 등으로 변경. `전설` 계열 삭제.

### 이슈명: 0 입력 결과 정책 충돌

- 심각도: **High**
- 해당 문서: `CALCULATION_RULES`, `UX_FLOW`, `IMPLEMENTATION_ROADMAP`
- 문제 설명: 0 입력 허용 여부와 빈 기록 리다이렉트 기준이 다름.
- 왜 문제인지: 라우팅, 저장, 테스트가 서로 다르게 구현될 수 있음.
- 수정 제안: `기록 없음`과 `0으로 저장된 기록`을 분리 정의.

### 이슈명: 회복시간의 의료/운전 오해 가능성

- 심각도: **High**
- 해당 문서: `CALCULATION_RULES`, `PRD`, `UX_FLOW`
- 문제 설명: `약 N시간`이 술이 깨는 시간처럼 보일 수 있음.
- 왜 문제인지: 운전 가능 여부로 오해될 위험.
- 수정 제안: `알코올 처리 추정 시간`으로 명명하고 `운전 판단 불가`를 가까이 표시.

### 이슈명: MVP 카드 범위 과다

- 심각도: **Medium**
- 해당 문서: `PRD`, `STORY_CARD_SPEC`, `TECHNICAL_PLAN`
- 문제 설명: 5종 템플릿 + 당일/주간 카드까지 MVP 포함.
- 왜 문제인지: 구현·QA·문구 안전성 검토 비용 증가.
- 수정 제안: MVP 2종, 나머지 확장.

### 이슈명: MVP 미구현 기능 stub 포함

- 심각도: **Medium**
- 해당 문서: `STORY_CARD_SPEC`, `TECHNICAL_PLAN`
- 문제 설명: 사용자 배경 업로드는 제외인데 `CustomBackgroundUploader` stub가 구조에 있음.
- 왜 문제인지: 구현자가 불필요한 파일/기능을 만들 가능성.
- 수정 제안: MVP 폴더 구조에서 제거.

## 5. 톤/문구 위험 요소

### 위험한 문구 목록

- `전설의 숙취술사`
  - 이유: 과음자를 멋있게 보이게 함.
  - 대체: `간 과부하 경보`

- `기록상은 전설`
  - 이유: 고위험 음주를 성취처럼 표현.
  - 대체: `기록상은 과부하, 회복은 미정`

- `자랑하기엔 좀 부족합니다`
  - 이유: 더 마시면 자랑 가능하다는 뉘앙스.
  - 대체: `공유할 숫자는 작고, 간 입장에선 다행입니다`

- `축하드립니다`
  - 이유: 과음 결과 축하로 읽힐 수 있음.
  - 대체: `알림입니다`

### 예외 허용 표현

- `소맥 견습생`
  - 이유: 사용자가 브랜드 톤상 예외로 유지하기로 결정.
  - 단, 주변 카피에서 `더 마시면 성장`, `다음 단계`, `숙련` 같은 게임화 표현은 피해야 한다.

## 6. 계산/티어 로직 수정 제안

### 모호한 기준

- 입력값 정수만 허용인지, 소주 0.5병 허용인지
- `0 기록`과 `기록 없음`의 데이터 차이
- 회복시간 표시명이 안전한지
- 주간 최다 음주일 동률 처리

### 테스트가 필요한 경계값

- 0
- 0.1
- 20 / 20.01
- 45 / 45.01
- 80 / 80.01
- 120 / 120.01
- 180 / 180.01
- 주간 80 / 200 / 400 / 700 / 1000 경계

### 추천 수정안

- 경계는 `0`, `>0 && <=20`, `>20 && <=45`처럼 정의.
- 회복시간은 `처리 추정 시간`으로 표시.
- 0 저장 기록은 결과 허용, 저장 기록이 전혀 없으면 리다이렉트.
- 높은 티어명에서 `전설/레전드/숙취술사` 제거.

## 7. UX/디자인 수정 제안

### 모바일 첫 화면

- 입력 카드와 결과 CTA가 첫 화면 하단 60% 안에 들어오게 유지.
- 안전 안내는 접히더라도 결과 화면에서는 반드시 보이게 함.

### 요일 선택 바

- 최근 7일 슬라이딩으로 통일.
- today, selected, recorded 상태를 명확히 분리.
- `이번 주`라는 표현은 최근 7일과 혼동되니 `최근 7일` 권장.

### 입력 카드

- 정수/소수 입력 정책 확정.
- 자동 저장 상태는 dot만으로 부족할 수 있어 접근성 label 필요.
- 30 max는 안전상 `입력 오류 방지용 상한`으로 내부 문서화.

### 결과 페이지

- `회복시간` 주변에 운전 판단 금지 안내 배치.
- 높은 티어일수록 공유 CTA 문구를 차분하게 유지.

### 주간 리포트

- `이번 주` 대신 `최근 7일`로 문구 통일.
- 최다 음주일 동률 표시 규칙 필요.

### 스토리 카드

- MVP 2종 권장: 기본 리포트, 간 야근 카드.
- `tpl_warning`은 확장으로 이동.
- 안전 안내 40자는 현재 고정 문구보다 짧게 별도 카드용 문구 필요.

## 8. 구현 전 반드시 확정해야 할 결정사항

- 최근 7일 슬라이딩 표현을 UI 전체에서 `최근 7일`로 통일할지
- 자동 저장만 둘지, 명시적 저장 버튼을 추가할지
- 소주/맥주 입력을 정수만 허용할지
- 0 입력 결과 페이지를 허용할지
- 기록 없음과 0 기록의 데이터 모델 차이
- 회복시간 명칭과 안내 문구
- 티어명 중 `전설/레전드` 제거 여부
- MVP 카드 템플릿 수
- 주간 스토리 카드 MVP 포함 여부
- 사용자 배경 업로드 stub 제거 여부
